# MQTT 数据丢失排查：400 台设备的 Client ID 冲突

## 背景

项目上有 400 台物联网设备，通过 MQTT 协议向服务器上报数据。整体架构很常见：

```
物联网设备 → MQTT Broker（服务器）→ 数据处理服务
```

设备出厂时会配置 MQTT 连接参数（Broker 地址、端口、Client ID 等），然后就自动连接上报。

## 问题现象

设备部署上去之后，数据一直丢。不是偶尔丢一条，是**经常性地丢数据**——有的设备上报了但服务器没收到，有的设备连上了但过一会儿就断开。

400 台设备，数据丢失没有规律，时好时坏。

## 排查过程

当时我刚开始接触 MQTT 协议，部门的老员工也没遇到过这种情况，没人能给现成答案。只能自己查文档 + 让 AI 帮忙总结排查思路。

### 第一步：排除网络问题

MQTT 数据丢失最常见的原因是网络不稳定。先从网络层排查：

1. **检查设备和服务器是否在同一网段** —— 确认是的
2. **从设备 ping 服务器** —— 能通，延迟正常
3. **从服务器 ping 设备** —— 能通
4. **检查网络稳定性** —— 长时间 ping 没有丢包

网络没问题。

### 第二步：查 MQTT Broker 日志

去服务器看了 MQTT Broker 的连接日志，发现一个奇怪的现象：

```
设备 A 连接成功
设备 A 断开连接
设备 A 连接成功
设备 A 断开连接
```

同一个设备反复连接、断开，而且断开的时间点和数据丢失的时间点完全吻合。

### 第三步：打开设备配置页面对比

既然网络没问题，那问题大概率出在 MQTT 连接本身。我打开了两台设备的配置页面，挨个参数对比：

```
┌─────────────────────────────────────────┐
│         设备 A 配置                       │
├─────────────────────────────────────────┤
│  Broker 地址：192.168.1.100              │
│  端口：1883                              │
│  Client ID：device_sensor_001            │
│  ...                                    │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│         设备 B 配置                       │
├─────────────────────────────────────────┤
│  Broker 地址：192.168.1.100              │
│  端口：1883                              │
│  Client ID：device_sensor_001            │
│  ...                                    │
└─────────────────────────────────────────┘
```

**Client ID 一模一样。**

## 根因

MQTT 协议规定，**同一个 Broker 上，Client ID 必须唯一**。当两台设备用相同的 Client ID 连接 Broker 时，后连接的会把先连接的踢掉，形成"反复连接、反复踢人"的死循环。

400 台设备里，有不少设备的 Client ID 是重复的。设备 A 连上去了，设备 B 用同一个 ID 连上来，A 被踢掉；A 重连，B 又被踢掉。数据就在这个过程中不断丢失。

## 解决方案

给每台设备分配唯一的 Client ID。最简单的做法是用设备的 MAC 地址或序列号作为 Client ID：

```
device_sensor_001  →  改为  →  device_00:1A:2B:3C:4D:5E
device_sensor_002  →  改为  →  device_00:1A:2B:3C:4D:5F
```

批量修改配置后重新部署，数据丢失问题消失。

## 总结

这个问题排查下来，最大的感触是：

1. **没人知道的时候，查文档是第一选择**。当时部门没人遇到过这个问题，我查了 MQTT 协议文档才搞清楚 Client ID 的规则
2. **对比法很有效**。网络排查没问题后，直接打开两台设备的配置页面逐项对比，一眼就看到了重复的 Client ID
3. **基础概念不能模糊**。当时刚接触 MQTT，对 Client ID 的作用理解不深，才导致设备出厂配置时没有做唯一性校验

> 后来整理了一篇 [MQTT 协议基础笔记](./MQTT协议基础.md)，把踩过的坑系统地记了下来。

**经验就是：部署前先校验关键配置的唯一性。遇到问题没人懂的时候，官方文档 + AI 辅助排查比瞎猜靠谱得多。**
