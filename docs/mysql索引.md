## 索引介绍

**索引是一种用于快速查询和检索数据的数据结构，其本质可以看成是一种排序好的数据结构。**

索引的作用就相当于书的目录。打个比方：我们在查字典的时候，如果没有目录，那我们就只能一页一页地去找我们需要查的那个字，速度很慢；如果有目录了，我们只需要先去目录里查找字的位置，然后直接翻到那一页就行了。

索引底层数据结构存在很多种类型，常见的索引结构有：B 树、 B+ 树 和 Hash、红黑树。在 MySQL 中，无论是 Innodb 还是 MyISAM，都使用了 B+ 树作为索引结构。

## 索引的优缺点

**索引的优点：**

1. **查询速度起飞 (主要目的)**：通过索引，数据库可以**大幅减少需要扫描的数据量**，直接定位到符合条件的记录，从而显著加快数据检索速度，减少磁盘 I/O 次数。
2. **保证数据唯一性**：通过创建**唯一索引 (Unique Index)**，可以确保表中的某一列（或几列组合）的值是独一无二的，比如用户 ID、邮箱等。主键本身就是一种唯一索引。
3. **加速排序和分组**：如果查询中的 ORDER BY 或 GROUP BY 子句涉及的列建有索引，数据库往往可以直接利用索引已经排好序的特性，避免额外的排序操作，从而提升性能。

**索引的缺点：**

1. **创建和维护耗时**：创建索引本身需要时间，特别是对大表操作时。更重要的是，当对表中的数据进行**增、删、改 (DML 操作)** 时，不仅要操作数据本身，相关的索引也必须动态更新和维护，这会**降低这些 DML 操作的执行效率**。
2. **占用存储空间**：索引本质上也是一种数据结构，需要以物理文件（或内存结构）的形式存储，因此会**额外占用一定的磁盘空间**。索引越多、越大，占用的空间也就越多。
3. **可能被误用或失效**：如果索引设计不当，或者查询语句写得不好，数据库优化器可能不会选择使用索引（或者选错索引），反而导致性能下降。

**那么，用了索引就一定能提高查询性能吗？**

**不一定。**大多数情况下，合理使用索引确实比全表扫描快得多。但也有例外：

- **数据量太小**：如果表里的数据非常少（比如就几百条），全表扫描可能比通过索引查找更快，因为走索引本身也有开销。
- **查询结果集占比过大**：如果要查询的数据占了整张表的大部分（比如超过 20%-30%），优化器可能会认为全表扫描更划算，因为通过索引多次回表（随机 I/O）的成本可能高于一次顺序的全表扫描。
- **索引维护不当或统计信息过时**：导致优化器做出错误判断。

## 索引底层数据结构选型

### Hash 表

哈希表是键值对的集合，通过键(key)即可快速取出对应的值(value)，因此哈希表可以快速检索数据（接近 O(1)）。

**为何能够通过 key 快速取出 value 呢？** 原因在于 **哈希算法**（也叫散列算法）。通过哈希算法，我们可以快速找到 key 对应的 index，找到了 index 也就找到了对应的 value。

```js
hash = hashfunc(key)
index = hash % array_size
```

**Hash 索引的缺点也很明显：**

- **不支持范围查询**：因为 Hash 表是无序的，`where age > 20` 这种查询会退化为全扫描。
- **不支持部分匹配**：联合索引必须用完整 key 才能查到，`where name like '张%'` 用不了。
- **Hash 冲突**：不同 key 算出相同 index 时，需要链地址法解决，性能下降。

所以 MySQL 的 InnoDB 只**在自适应哈希索引 (Adaptive Hash Index) 场景下**用 Hash，默认的索引结构是 B+ Tree。

### B+ Tree 索引

> 💡 **搞懂 B+ Tree 很重要，它能帮你理解为什么 MySQL 的索引"长这样"。**

#### B+ Tree 长什么样？

假设我们有一个用户表，在 `age` 列上建了索引：

```sql
select * from user where age = 22;
```

B+ Tree 索引的结构可以简化为三层：

```
                      [50, 100]                    ← 内节点（仅存 key，不存数据）
                     /     |    \
                    /      |     \
            [10,30,50]  [60,80,100]  [120,150]     ← 内节点
              /    |   \     ...        ...
             /     |    \
        (data) (data) (data)              ← 叶子节点（存 key + 数据/指针）
```

**核心特点：**

1. **只有叶子节点存数据**，非叶子节点只存 key，用来"指路"。
2. **叶子节点之间有双向链表连接**，方便范围查询时直接向后遍历。
3. **一个节点就是一页（16KB）**，InnoDB 以页为单位读写磁盘。

#### 为啥 MySQL 选 B+ Tree 而不是别的？

| 数据结构 | 为什么不选 |
|---------|-----------|
| **Hash** | 不支持范围查询、不支持排序、不支持部分匹配 |
| **红黑树** | 树太高了。2000 万条数据 ≈ 25 层，每次查询要 25 次磁盘 I/O |
| **B 树** | 内节点也存数据，一页能存的 key 太少，树变高；且范围查询需要中序遍历，效率低 |
| **B+ Tree** | 非叶子节点只存 key，一页能存上千个 key，2000 万条数据**3 层就够了** |

#### 计算一下：为什么 3 层 B+ Tree 能存 2000 万条？

- 一页 16KB，假设一个 bigint key 8B + 指针 6B = 14B
- 一个内节点可存：`16384 / 14 ≈ 1170 个 key`
- 叶子节点一行数据约 1KB，一页可存 16 条
- **总数据量 ≈ 1170 × 1170 × 16 ≈ 2190 万条**

也就是说：**2000 万条数据，3 次磁盘 I/O 就能查到**，这就是 B+ Tree 的恐怖之处。

## 索引的分类

### 按数据结构分

| 类型 | 说明 |
|------|------|
| **B+ Tree 索引** | InnoDB 默认，几乎所有场景都用它 |
| **Hash 索引** | InnoDB 的自适应哈希索引（自动管理），Memory 引擎支持 |
| **全文索引** | 用于大文本的 `like '%xxx%'` 和 `match ... against` 场景 |

### 按存储方式分（聚簇 vs 非聚簇）

这个比较重要，单独拿出来说 ⬇️

### 按功能/逻辑分

| 类型 | 说明 | 关键字 |
|------|------|--------|
| **主键索引** | 特殊的唯一索引，不允许为 null，一张表只能有一个 | `primary key` |
| **唯一索引** | 索引列的值必须唯一，允许 null | `unique` |
| **普通索引** | 仅加速查询，不要求唯一性 | `index` |
| **联合索引** | 多列组合成一个索引，遵循最左前缀原则 | `index(a,b,c)` |
| **全文索引** | 解决 like '%word%' 性能问题 | `fulltext` |

## 聚簇索引 & 非聚簇索引

这是 InnoDB 和 MyISAM 最核心的区别。

### 聚簇索引（InnoDB）

**聚簇索引的意思是：索引和数据存在一起。**

```
┌─────────────────────────────────────────────────────────────┐
│                    InnoDB 聚簇索引结构                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   B+ Tree 叶子节点 (主键索引 id)                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  ┌────┬──────────────────────────────────┐         │   │
│   │  │ id │           完整行数据               │         │   │
│   │  ├────┼──────────────────────────────────┤         │   │
│   │  │ 1  │ name=张三, age=18, city=北京     │ ◀── 叶子 │   │
│   │  ├────┼──────────────────────────────────┤         │   │
│   │  │ 2  │ name=李四, age=22, city=上海     │ ◀── 节点 │   │
│   │  ├────┼──────────────────────────────────┤  存储整  │   │
│   │  │ 3  │ name=王五, age=25, city=广州     │ ◀── 行数据│   │
│   │  ├────┼──────────────────────────────────┤         │   │
│   │  │... │          ...                     │         │   │
│   │  └────┴──────────────────────────────────┘         │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   非叶子节点只存 key + 指针，用来"指路"，不存行数据            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

结论：找到了索引就找到了数据，无需回表
```

**InnoDB 的聚簇规则：**

| 优先级 | 条件 | 聚簇索引 |
|--------|------|----------|
| 1 | 有主键 | 主键作为聚簇索引 |
| 2 | 没有主键，但有唯一非空列 | 选它作为聚簇索引 |
| 3 | 都没有 | InnoDB 自动生成 6 字节的 `row_id` |

### 非聚簇索引（MyISAM / InnoDB 的辅助索引）

**非聚簇索引的意思：索引和数据分开存。**

```
┌──────────────────────────────────────────────────────────────────┐
│                    InnoDB 辅助索引 + 回表过程                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   辅助索引 (age)              主键索引 (id)         聚簇索引(数据)  │
│   ┌─────────────────┐       ┌──────────────┐     ┌────────────┐ │
│   │ age │  主键id    │       │  id          │     │   行数据    │ │
│   ├─────┼───────────┤       ├──────────────┤     ├────────────┤ │
│   │ 18  │ ──────── 2│──────▶│  2           │────▶│  李四,22,沪│ │
│   ├─────┼───────────┤       ├──────────────┤     ├────────────┤ │
│   │ 20  │ ──────── 3│──┐    │  3           │────▶│  王五,25,穗│ │
│   ├─────┼───────────┤  │    ├──────────────┤     ├────────────┤ │
│   │ 22  │ ──────── 1│──┼───▶│  1           │────▶│  张三,18,京│ │
│   └─────┴───────────┘  │    └──────────────┘     └────────────┘ │
│                        │                                         │
│   叶子节点只存主键值     └── 第二次回表查询（随机I/O，较慢）        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**核心概念：**

| 概念 | 解释 | 示例 |
|------|------|------|
| **回表** | 走辅助索引查到主键 id，再到聚簇索引查完整数据 | `select * from user where age = 22` 先查 age 索引拿到 id=1，再回主键索引拿 name 等 |
| **覆盖索引** | 查询的列全部在辅助索引里，无需回表 | `select age from user where age > 18`，age 索引里有 age，直接返回 |
| **索引下推 (ICP)** | MySQL 5.6+，在索引层面先过滤，减少回表次数 | 联合索引 `(name, age)`，`where name like '张%' and age = 18`，在索引里直接判断 age |

## 联合索引 & 最左前缀原则

### 什么是联合索引？

联合索引就是**对多个列同时建一个索引**：

```sql
create index idx_name_age on user(name, age);
```

### 最左前缀原则

MySQL 会**从联合索引的最左边开始匹配**，直到遇到范围查询（`>`, `<`, `between`, `like`）就停止匹配。

```sql
-- 假设有联合索引 (a, b, c)

where a = 1            -- ✅ 用到索引 (a)
where a = 1 and b = 2  -- ✅ 用到索引 (a, b)
where a = 1 and b = 2 and c = 3  -- ✅ 全部用到
where b = 2            -- ❌ 索引失效，跳过了 a
where c = 3            -- ❌ 索引失效
where a = 1 and c = 3  -- ⚠️ 只用到了 a，c 用不了（跳过了 b）
```

**口诀：带头大哥不能死，中间兄弟不能断。**

### 联合索引的排序规则

联合索引 `(a, b)` 的排序逻辑是：**先按 a 排，a 相同再按 b 排**。这就意味着：

```sql
where a = 1 order by b  -- ✅ 索引排序，不用 filesort
where a = 1 order by c  -- ❌ 索引无法排序，需要 filesort
```

## 索引失效的常见场景

这是**面试必问 + 日常必踩**的部分，建议收藏。

### 1. 联合索引违反最左前缀

```sql
-- 索引 (a, b, c)
where b = 1           -- ❌ 跳过 a
where a = 1 and c = 2 -- ⚠️ 只用 a，c 用不了
```

### 2. 对索引列使用函数或运算

```sql
where age + 1 = 20   -- ❌ 索引列参与了运算
where DATE(create_time) = '2024-01-01'  -- ❌ 对索引列用了函数

-- ✅ 正确写法
where age = 19
where create_time >= '2024-01-01' and create_time < '2024-01-02'
```

### 3. 隐式类型转换

如果 `phone` 列是 varchar，但传了数字：

```sql
where phone = 13800138000  -- ❌ 类型不匹配，MySQL 会做隐式转换
                            -- 相当于 CAST(phone AS signed int)，索引列用了函数

where phone = '13800138000' -- ✅ 类型匹配
```

### 4. 使用 % 开头的 LIKE

```sql
where name like '%张三'   -- ❌ 以 % 开头，无法利用 B+ Tree 的排序
where name like '张三%'   -- ✅ 前缀匹配，可以用索引
```

### 5. OR 条件中存在无索引列

```sql
-- 假设 name 有索引，age 没有索引
where name = '张三' or age = 18  -- ❌ age 无索引，整个 OR 无法用索引

-- ✅ 用 union 拆开
where name = '张三'
union
where age = 18

-- ✅ 或者给 age 也加上索引
```

### 6. 使用 !=, <>, NOT IN, NOT EXISTS

```sql
where age != 18     -- ❌ 不等于无法走索引（除非数据分布极特殊）
where age <> 18     -- ❌ 同上
where id not in (1,2,3)  -- ❌
```

例外：如果 `!= 'xxx'` 筛选掉的是极少量数据，优化器可能还是选索引，但**不要依赖这个**。

### 7. IS NULL / IS NOT NULL

```sql
where name is null     -- ✅ 可以走索引（InnoDB 会把 null 值集中放在索引一边）
where name is not null -- ⚠️ 如果大部分数据都不为 null，优化器可能放弃索引
```

### 8. 数据分布导致优化器放弃索引

```sql
-- 表里有 100 万条数据，gender 只有 '男' 和 '女'
-- 如果 gender 上建了索引：

where gender = '男'  -- ⚠️ 优化器可能直接全表扫描
                     -- 因为要返回的数据量太大（≈50%），回表成本高于全表扫描
```

## 索引优化建议

### 1. 选择合适的列建索引

- **经常出现在 where 条件的列** → 建索引
- **经常 order by / group by 的列** → 建索引
- **经常 join 的列** → 建索引
- **频繁更新的列** → 谨慎建索引（维护成本高）
- **区分度低的列**（性别、状态码等） → 不建议单列索引，可考虑联合索引

### 2. 合理设计联合索引

- 把**区分度高的列放前面**
- 把**经常等值查询的列放前面**
- 把**经常范围查询的列放后面**
- 尽量通过联合索引实现**覆盖索引**，避免回表

```sql
-- 经常这样查
select name, age from user where name = '张三' and age > 18;

-- ✅ 索引设计 (name, age)
-- name 放前面（等值查询），age 放后面（范围查询）
-- 且查询的 name 和 age 都在索引里，覆盖索引，无需回表
```

### 3. 善用 explain 分析

```sql
explain select * from user where name = '张三';
```

重点关注：
- **`type`**：const > ref > range > index > ALL（从好到差）
- **`key`**：实际用的索引
- **`rows`**：扫描行数（越少越好）
- **`Extra`**：Using index（覆盖索引✅），Using filesort（需要额外排序❌）

### 4. 避免冗余索引

```sql
-- 已经有索引 (a, b)，再建 (a) 是冗余的，因为 (a, b) 已经能覆盖 a 的查询
```

## 索引创建语法速查

### 创建索引

```sql
-- 直接创建
CREATE INDEX idx_name ON user(name);

-- 创建表时创建
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    INDEX idx_name (name)
);

-- 联合索引
CREATE INDEX idx_name_age ON user(name, age);

-- 唯一索引
CREATE UNIQUE INDEX idx_phone ON user(phone);
```

### 删除索引

```sql
-- 删除普通索引
DROP INDEX idx_name ON user;

-- 删除主键索引
ALTER TABLE user DROP PRIMARY KEY;
```

### 查看索引

```sql
-- 查看表的所有索引
SHOW INDEX FROM user;
```

## 慢查询优化实战

### 场景：订单表查询慢

假设有一张 `orders` 表，数据量 500 万，有一条慢查询：

```sql
SELECT * FROM orders WHERE user_id = 10086 AND status = 'paid' ORDER BY create_time DESC LIMIT 10;
```

**第一步：EXPLAIN 分析**

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 10086 AND status = 'paid' ORDER BY create_time DESC LIMIT 10;
```

结果：

| type | key | rows | Extra |
|------|-----|------|-------|
| ALL | NULL | 5000000 | Using filesort |

问题：全表扫描 + filesort，非常慢。

**第二步：建立索引**

```sql
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);
```

索引设计思路：
- `user_id` 放最前面：等值查询，区分度高
- `status` 放中间：等值查询
- `create_time` 放最后：ORDER BY 排序，利用索引有序性

**第三步：再次 EXPLAIN**

| type | key | rows | Extra |
|------|-----|------|-------|
| ref | idx_user_status_time | 10 | Using index |

优化效果：扫描行数从 500 万降到 10 行，消除 filesort，查询从秒级降到毫秒级。

### 优化口诀

```
全表扫描加索引，索引设计有讲究：
等值查询放前面，范围查询放后头，
排序字段加末尾，覆盖索引不用愁。
```

## 索引相关面试题

**Q：为什么 MySQL 用 B+ Tree 不用 B Tree？**

A：B+ Tree 的非叶子节点不存数据，一页能存更多 key，树更矮；叶子节点有链表连接，范围查询直接往后遍历就行，而 B Tree 需要中序遍历。

**Q：什么是回表？怎么避免？**

A：走辅助索引查到主键 id，再回聚簇索引查完整数据。用**覆盖索引**可以避免回表。

**Q：联合索引 (a, b) 和 (b, a) 一样吗？**

A：不一样。排序规则不同，(a, b) 先按 a 排再按 b 排，(b, a) 反过来。这决定了 `where b = 1 order by a` 能用哪个索引。

**Q：索引建的越多越好吗？**

A：不是。索引会降低写入性能、占用磁盘空间、维护成本高。一般单表索引控制在 5 个以内。

**Q：针对大表，如何优化慢查询？**

A：先 explain 看是否走了索引 → 没走索引就加合适的索引（考虑联合索引、覆盖索引）→ 走了索引还慢就考虑分库分表。
