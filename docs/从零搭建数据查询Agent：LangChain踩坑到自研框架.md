# 从零搭建数据查询 Agent：LangChain 踩坑到自研框架

## 背景

项目甲方要求系统集成 AI 能力，能用自然语言查询业务数据。当时部门没有 AI 相关的人才，而我是智能科学与技术专业出身，这块正好是我的知识储备范围，所以这个任务就落到我头上了。

需求很明确：**用户输入自然语言 → AI 理解意图 → 生成 SQL → 查询数据库 → 返回结果**。

## 技术选型

### 为什么用 Python

整个项目后端是 Java 体系，但 AI 这块必须用 Python，原因有两个：

1. **生态**：LangChain、vLLM 这些主流 AI 框架都是 Python 优先
2. **本地部署需求**：有的厂家要求数据安全，不能联网调 API，必须本地部署大模型。本地部署方案（vLLM）天然就是 Python 生态

最终采用**微服务架构**，前端把 AI 相关的请求代理到 Python 服务，Java 服务负责业务逻辑，两边各司其职：

```
前端请求
  ├── 业务接口 → Java 服务
  └── AI 对话  → Python 服务（Agent + vLLM）
```

### 为什么选 LangChain

当时 LangChain 是最火的 AI 应用框架，文档多、社区活跃，看起来是"标准方案"。就直接用了。

## 实现方案

### 整体架构

```
用户输入自然语言
    ↓
LangChain Agent（理解意图）
    ↓
调用工具函数（生成 SQL、查询数据库）
    ↓
返回结果 + 推荐图表
```

### RAG 部分

为了让 AI 能理解业务术语（比如"良率"、"OEE"这些专业词），用 FAISS 向量数据库做了一个 RAG：

- 把业务术语表、表结构说明、常见问题等文档向量化
- 用户提问时先检索相关文档，拼到 Prompt 里
- AI 结合上下文理解业务含义，再生成 SQL

## 踩坑：上下文窗口超限

项目用的是本地部署的 8G 显存版本模型，上下文窗口本身就不大。

LangChain 的 Agent 加载工具函数时，会把所有工具的描述都塞到系统提示词里。当工具函数加到 **6 个**的时候，上下文直接爆了——工具描述 + 用户提问 + RAG 检索结果 + 对话历史，总 token 数超出了模型的上下文限制。

表现就是：模型开始胡言乱语，或者直接截断输出。

### 解决方案

当时想了两个方向：

1. **逐渐披露工具函数**：不一次性把所有工具都塞给模型，而是先用一个"路由"判断用户意图，只加载相关的工具
2. **多 Agent 拆分**：把一个大 Agent 拆成多个小 Agent，每个 Agent 负责一个领域，各自只加载自己的工具

> 当时还没有"Skills"这种概念，全靠自己想办法。

最终用的是**逐渐披露**的方案——先判断用户想查什么类型的数据，再动态加载对应的工具函数，避免一次性塞太多工具描述。

## 踩坑：LangChain 的"重量感"

用 LangChain 开发的过程中越来越觉得别扭：

- **抽象层太多**：想改一个底层行为，要翻好几层源码
- **调试困难**：出错的时候堆栈信息被层层包装，定位问题很痛苦
- **灵活性不够**：需求稍微偏离框架设计，就要 hack 很多东西

后来离职后，我自己写了一个轻量的 Agent 框架，不用 LangChain，核心逻辑就几百行代码：[DataCopilot](https://gitee.com/khc-warehouse/data-copilot)

```
理解意图 → 选择工具 → 执行 → 观察结果 → 决定下一步
```

这个循环本身并不复杂，LangChain 把它包装得太重了。

## 最终效果

| 指标 | 数据 |
|------|------|
| 支持的查询类型 | 销售、生产、质量等多维度 |
| 响应时间 | 2-5s（含模型推理） |
| SQL 准确率 | 80%+（有 RAG 辅助） |
| 部署方式 | 本地 vLLM，满足数据安全要求 |

## 总结

这个项目从零开始搭了一套 AI 数据查询系统，踩了不少坑，但也积累了不少经验：

1. **本地部署大模型要量力而行**：8G 显存的限制不只是推理速度，上下文窗口也是瓶颈
2. **工具函数不要贪多**：Agent 加载太多工具反而效果差，按需加载更靠谱
3. **框架不是越重越好**：LangChain 适合快速原型，但生产环境可能需要更轻量的方案
4. **RAG 是本地部署的刚需**：没有联网能力，业务知识只能靠 RAG 注入
