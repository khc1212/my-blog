# 接口性能优化：从 2 秒到 200 毫秒

## 背景

项目上有个设备数据查询接口，前端页面打开就要调，加载设备列表的同时还要展示每个设备的最新一条数据。

最初开发的时候功能能用就没管，后来设备越来越多，接口响应时间从几百毫秒涨到了 **2 秒以上**，前端页面白屏等待，甲方开始抱怨。

## 原始实现

接口逻辑大概是这样的：

```java
@GetMapping("/devices")
public Result getDeviceList() {
    // 1. 查所有设备
    List<Device> devices = deviceMapper.selectAll();
    
    List<DeviceVO> result = new ArrayList<>();
    for (Device device : devices) {
        // 2. 每个设备查最新一条数据（N+1 问题）
        DeviceData latest = deviceDataMapper.selectLatestByDeviceId(
            device.getDeviceId()
        );
        
        DeviceVO vo = new DeviceVO();
        vo.setDeviceId(device.getDeviceId());
        vo.setName(device.getName());
        vo.setLatestData(latest);
        result.add(vo);
    }
    
    return Result.success(result);
}
```

```xml
<!-- DeviceDataMapper.xml -->
<select id="selectLatestByDeviceId" resultType="DeviceData">
    SELECT * FROM device_data
    WHERE device_id = #{deviceId}
    ORDER BY ts DESC
    LIMIT 1
</select>
```

看起来没问题，但魔鬼在细节里。

## 问题分析

### 问题一：N+1 查询

100 台设备 = 1 次查设备列表 + 100 次查最新数据 = **101 次 SQL 查询**。

每次查询都要走一次数据库连接、解析、执行、返回，100 台设备就是 100 次往返。

### 问题二：没有索引

`device_data` 表按 `device_id` + `ts` 排序查最新数据，但 `device_id` 字段没加索引（又是这个问题），每次查询都是全表扫描。

### 问题三：每次都查数据库

设备信息和最新数据变化频率其实不高，但每次接口请求都查数据库，完全没有缓存。

## 优化方案

### 第一步：加索引

先解决最基础的问题：

```sql
ALTER TABLE device_data ADD INDEX idx_device_ts (device_id, ts);
```

联合索引 `(device_id, ts)` 直接覆盖了 `WHERE device_id = ? ORDER BY ts DESC` 的查询模式。

### 第二步：消除 N+1

用一条 SQL 替代循环查询：

```xml
<!-- DeviceDataMapper.xml -->
<select id="selectLatestByDeviceIds" resultType="DeviceData">
    SELECT t1.*
    FROM device_data t1
    INNER JOIN (
        SELECT device_id, MAX(ts) AS max_ts
        FROM device_data
        GROUP BY device_id
    ) t2 ON t1.device_id = t2.device_id AND t1.ts = t2.max_ts
    WHERE t1.device_id IN
    <foreach collection="deviceIds" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>
```

```java
@GetMapping("/devices")
public Result getDeviceList() {
    List<Device> devices = deviceMapper.selectAll();
    List<String> deviceIds = devices.stream()
        .map(Device::getDeviceId)
        .collect(Collectors.toList());
    
    // 一次查询拿到所有设备的最新数据
    Map<String, DeviceData> latestMap = deviceDataMapper
        .selectLatestByDeviceIds(deviceIds)
        .stream()
        .collect(Collectors.toMap(
            DeviceData::getDeviceId, 
            d -> d
        ));
    
    List<DeviceVO> result = devices.stream().map(device -> {
        DeviceVO vo = new DeviceVO();
        vo.setDeviceId(device.getDeviceId());
        vo.setName(device.getName());
        vo.setLatestData(latestMap.get(device.getDeviceId()));
        return vo;
    }).collect(Collectors.toList());
    
    return Result.success(result);
}
```

**101 次查询 → 2 次查询**。

### 第三步：加缓存

设备列表和最新数据不需要实时性特别高，缓存 10 秒完全可接受：

```java
@GetMapping("/devices")
public Result getDeviceList() {
    String cacheKey = "device_list_with_latest";
    
    // 先查 Redis 缓存
    Object cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return Result.success(cached);
    }
    
    // 缓存没命中，查数据库
    List<DeviceVO> result = queryDeviceListWithLatest();
    
    // 写入缓存，10 秒过期
    redisTemplate.opsForValue().set(cacheKey, result, 10, TimeUnit.SECONDS);
    
    return Result.success(result);
}
```

10 秒内重复请求直接走 Redis，数据库压力骤降。

### 第四步：分页

设备多了之后，一次返回几百条数据本身就不合理。用 MyBatis-PageHelper 或者手动分页：

```java
@GetMapping("/devices")
public Result getDeviceList(
        @RequestParam(defaultValue = "1") int page,
        @RequestParam(defaultValue = "20") int pageSize) {
    
    // 分页查询设备列表
    int offset = (page - 1) * pageSize;
    List<Device> devices = deviceMapper.selectByPage(offset, pageSize);
    int total = deviceMapper.countAll();
    
    // 查这批设备的最新数据
    List<String> deviceIds = devices.stream()
        .map(Device::getDeviceId)
        .collect(Collectors.toList());
    Map<String, DeviceData> latestMap = deviceDataMapper
        .selectLatestByDeviceIds(deviceIds)
        .stream()
        .collect(Collectors.toMap(DeviceData::getDeviceId, d -> d));
    
    // 组装返回
    List<DeviceVO> list = devices.stream().map(device -> {
        DeviceVO vo = new DeviceVO();
        vo.setDeviceId(device.getDeviceId());
        vo.setName(device.getName());
        vo.setLatestData(latestMap.get(device.getDeviceId()));
        return vo;
    }).collect(Collectors.toList());
    
    PageResult<DeviceVO> result = new PageResult<>();
    result.setTotal(total);
    result.setPage(page);
    result.setPageSize(pageSize);
    result.setData(list);
    
    return Result.success(result);
}
```

每次只返回 20 条，前端按需加载。

## 优化效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| SQL 查询次数 | 101 次 | 2 次 |
| 响应时间 | 2-3s | 200-300ms |
| 走缓存时 | - | < 50ms |
| 数据传输量 | 全量 | 分页 20 条 |

## 总结

接口优化就这几招，按优先级排：

1. **加索引**：成本最低，效果最明显，SQL 慢先看有没有索引
2. **消除 N+1**：循环里查数据库是大忌，MyBatis 里尤其容易犯
3. **加缓存**：读多写少的数据别每次都查库，Redis 一挡大部分请求
4. **分页**：数据多了就分批返回，别一口气全吐出来

> 相关笔记：[MySQL 索引](/mysql索引)、[MySQL 日志](/mysql日志) 记录了索引和查询优化的基础知识。

实际项目中这四个方案通常是组合使用的，不是非此即彼。先加索引解决最紧迫的问题，再优化查询逻辑，最后加缓存兜底。
