# 物联网设备掉线排查：MySQL 慢查询引发的连锁反应

## 背景

项目上有一套物联网数据采集系统，主要两个模块：

- **设备信息管理模块**：管理所有物联网设备的基础信息（设备 ID、类型、状态等）
- **数据采集模块**：接收设备上报的数据，写入时序数据库

整体数据流很简单：

```
物联网设备 → 数据采集服务 → MySQL 查设备信息 → 写入时序数据库
```

设备上报数据时会携带一个 `device_id`，采集服务需要先根据这个 ID 去 MySQL 查设备基础信息，校验通过后才把数据写入时序数据库。

## 问题现象

设备数量上来之后（每秒至少 100 条上报），开始频繁出现设备"掉线"——实际上设备本身没掉线，是采集服务查设备信息超时，导致上报数据被丢弃，业务上看起来就像设备掉线了。

刚开始只是偶尔掉几台，后来随着设备越来越多，掉线越来越频繁，甚至出现大面积设备离线的情况。

## 排查过程

### 第一步：看日志

先看了采集服务的日志，发现大量这样的报错：

```
查询设备信息超时，device_id: xxx
```

初步定位到是查询设备信息这一步出了问题。

### 第二步：看 MySQL 慢查询

去 MySQL 开了慢查询日志，发现这条 SQL 赫然在列：

```sql
SELECT * FROM device_info WHERE device_id = 'xxx';
```

一条简单的等值查询，执行时间居然要好几秒。设备少的时候没感觉，每秒 100+ 次并发查询直接把 MySQL 打趴了。

### 第三步：分析原因

看了下 `device_info` 表的结构，发现 `device_id` 字段**没有加索引**。

这意味着每次查询都是全表扫描。表里设备信息不多（几千条），单条查询可能也就几十毫秒，但在高并发下，大量查询排队等待，连接池被打满，最终导致查询超时。

用 `EXPLAIN` 验证了一下：

```sql
EXPLAIN SELECT * FROM device_info WHERE device_id = 'xxx';
```

```
+----+-------------+-------------+------+---------------+------+---------+------+-------+-------------+
| id | select_type | table       | type | possible_keys | key  | key_len | ref  | rows  | Extra       |
+----+-------------+-------------+------+---------------+------+---------+------+-------+-------------+
|  1 | SIMPLE      | device_info | ALL  | NULL          | NULL | NULL    | NULL |  3847 | Using where |
+----+-------------+-------------+------+---------------+------+---------+------+-------+-------------+
```

`type: ALL`，全表扫描实锤。

## 根因

**`device_id` 字段没有索引，导致每次设备上报都要全表扫描。**

设备少的时候问题不明显，一旦并发上来（每秒 100+ 查询），MySQL 扛不住，查询超时，设备上报数据被丢弃，业务上表现为设备掉线。

## 解决方案

### 1. 加索引

第一步是给 `device_id` 加索引：

```sql
ALTER TABLE device_info ADD INDEX idx_device_id (device_id);
```

加完索引后，同样的查询：

```
+----+-------------+-------------+------+---------------+---------------+---------+-------+------+-------+
| id | select_type | table       | type | possible_keys | key           | key_len | ref   | rows | Extra |
+----+-------------+-------------+------+---------------+---------------+---------+-------+------+-------+
|  1 | SIMPLE      | device_info | ref  | idx_device_id | idx_device_id | 128     | const |    1 |       |
+----+-------------+-------------+------+---------------+---------------+---------+-------+------+-------+
```

`type: ref`，直接走索引，查询时间从秒级降到毫秒级。

> 这次踩坑之后，专门整理了一篇 [MySQL 索引](/mysql索引) 的笔记，把索引的原理和使用场景系统地过了一遍。

### 2. 加 Redis 缓存

索引解决了查询效率问题，但每秒 100+ 次查询对 MySQL 还是有压力。设备基础信息变化频率很低，没必要每次都查数据库。

加了一层 Redis 缓存：

```
设备上报 → 先查 Redis → 命中则直接用
                    → 未命中则查 MySQL → 写入 Redis（设置过期时间）
```

设备信息缓存到 Redis 后，绝大多数请求直接走缓存，MySQL 的查询压力降了一个数量级。

## 最终效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 设备信息查询耗时 | 2-5s | < 1ms |
| 每秒处理上报数 | 不稳定，频繁超时 | 稳定 200+ |
| 设备掉线率 | 频繁 | 几乎为零 |

## 总结

这个问题的本质很简单——**没加索引**。但排查过程暴露了几个问题：

1. **表设计阶段没考虑查询场景**：`device_id` 作为高频查询条件，建表时就应该加索引
2. **缺乏监控**：慢查询日志没开，问题积累到设备大面积掉线才发现
3. **缓存策略缺失**：设备信息这种读多写少的数据，应该一开始就加缓存

经验就是：**上线前先用 `EXPLAIN` 看看 SQL 执行计划，高频查询的字段一定要有索引。读多写少的数据，尽早加缓存。**

这次事故之后，专门把 MySQL 索引、日志、锁这些知识点系统地补了一遍，笔记整理在了 [MySQL 索引](/mysql索引)、[MySQL 日志](/mysql日志)、[MySQL 锁](/mysql锁)。踩坑不可怕，可怕的是同一个坑踩两次。
